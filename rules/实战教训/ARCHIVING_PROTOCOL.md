# 新增案例归档协议（bot 自动执行）

> 本文件是**机器人自动归档新复盘案例的执行规范**。人类复盘或 bot 自动复盘产出一个新案例后，
> 严格按本协议的「三写一改」四步落盘，无需人工介入。
> 面向对象：在服务器上自动总结并写库的 LLM bot。执行前必读本文件全文。

---

## 一、前置：读取当前编号（防止编号冲突）

落盘前先扫描 `rules/实战教训/` 目录，确定两个下一号：

1. **案例号 N**：读 `reference_case_lessons.md`「案例索引」表最后一行的 `#`，+1。
2. **数据卡号 XX**：`ls` 目录下 `20260*_case_*.md`，取文件名中最大的 `case_XX`，+1。

> ⚠️ 案例号（索引里的 #）与数据卡号（文件名里的 case_XX）**是两套独立序列**，不一定相等。
> 例：截至案例 26，数据卡才到 case_06（因早期多场无数据存档）。**分别 +1，不要混用**。

---

## 二、路由：判断新案例归入哪个主题文件（唯一需要判断的一步）

对照 `reference_case_lessons.md`「主题路由表」，按新案例的**核心教训**匹配最贴合的现有主题文件。判定顺序：

1. **优先归入现有主题**：核心教训能对上某个 `feedback_*.md` 的主题 → 在该文件内**追加一节**（用下一个大写字母编号：已有 A/B → 新增 C）。
2. **一个案例可挂多个主题**：若教训横跨多主题（如既是凯利信号又是中超基本面），在每个相关文件补规则，索引「教训主题」列写多个（用 `；` 分隔，如 `kelly F；csl D`）。
3. **确实无处可归才新建**：只有当核心教训不属于任何现有 10 个主题时，才新建 `feedback_<主题>.md`，并在「主题路由表」加一行触发条件。**这是例外，不是常态**——新建前先确认真的挂不上任何现有主题。

> 路由判断示例（本次 Brazil-Norway）：核心教训是「世界杯强弱对话中受让方遇特定信号要改判不败优先」，
> 与 `strong_team_deep`（强队深盘/杯赛冷平）B 节「受让方可直接反胜」同源 → 归入该文件，新增 C 节。

---

## 三、执行「三写一改」

### ① 写数据卡 `20260XXXX_case_XX_<home>_vs_<away>.md`

- 命名：`日期YYYYMMDD_case_两位序号_主队英文_vs_客队英文.md`（队名小写、下划线连接，与源数据一致）
- 顶部引用行格式（**必须**，指向主题文件的具体节 + 索引案例号）：
  ```
  > 数据存档卡。完整论证见 [feedback_<主题>.md](feedback_<主题>.md) <节> 节（<一句话主题>）。案例索引见 [reference_case_lessons.md](reference_case_lessons.md) 案例 <N>。
  ```
- 固定四段结构：`## 结果` / `## 关键信号` / `## 教训与规则` / `## 衍生规则`
- `## 结果` 三小节：**预测**（原判，标注对/错/偏差）、**实际**（真实赛果）、**正确结论**（复盘后应得的结论）
- `## 衍生规则` 尾注：一行写明本案催生了哪个主题文件的哪节

### ② 写规则（在路由命中的主题文件内追加一节）

- 用下一个字母编号（A/B 已存在 → 新增 `## C. <节标题>`），插在最后一个规则节之后、`## 本主题检查项` 之前。
- 节内**首行**必须是来源引用（含日期、赛事、错在哪、存档链接）：
  ```
  > 来源：<对阵>（<日期> <赛事>）。<原判错误一句话>。存档：[20260XXXX_case_XX_xxx.md](20260XXXX_case_XX_xxx.md)。
  ```
- 若与同文件既有节是姊妹场景，加一段「与 A/B 的区别」说明边界，避免规则重叠混淆。
- 规则用 `**规则 C1：...**`、`**规则 C2：...**` 编号，含阈值/条件表格更佳。

### ③ 改检查项（同一主题文件的「本主题检查项」）

- 为新增的每条规则补一条 `- [ ]` 勾选项，句式：`触发条件？→ 应做的动作`。
- 若新案例扩展了该文件的**触发边界**，同步更新文件顶部 `> **触发条件**：` 行与「相关」节的案例列表。

### ④ 改索引（`reference_case_lessons.md`）

- **案例索引表末尾追加一行**（新案例始终在末尾，即使日期早于既有——编号只增不插）：
  ```
  | N | 主队 比分 客队 | YYYY-MM-DD | 赛事 | 赛果一句话 | 主题缩写 节 | [数据卡链接] |
  ```
- **若新增/扩展了触发条件**：同步更新「主题路由表」对应行的「触发条件」与「核心要点」列（否则 bot 下次遇到该盘型不会命中此文件——本步骤最易漏，务必检查）。
- 若新建了主题文件：在「主题路由表」加一整行，并在表下方「主题缩写对应文件」注释补一项。

---

## 四、落盘后自检（提交前必过）

- [ ] 案例号、数据卡号均在原最大值 +1，无冲突、无跳号
- [ ] 数据卡顶部引用的「主题文件 + 节 + 案例号」与实际写入位置**三方一致**
- [ ] 主题文件新增节的字母编号连续（不跳，不重复）
- [ ] 「本主题检查项」已为新规则补勾选项
- [ ] 若动了触发边界：主题文件顶部触发条件 + 总览路由表触发条件**两处都改了**
- [ ] 索引表新行的「主题缩写 节」与主题文件实际节号对得上
- [ ] 全库无残留裸章节号（旧版「第X章」式引用一律用「文件名 + 字母节」）：
      `grep -rnE "第[一二三四五六七八九十]+[、~-]?章" rules/实战教训/2026*_case_*.md` 应为空

---

## 五、提交

- 暂存范围：仅 `rules/实战教训/`（含新数据卡、改动的主题文件、总览）。
- **不要**暂存 `.claude/`、`report/`、`data/` 等无关或 gitignore 目录。
- commit message 格式：`新增案例N: <对阵> — <一句话教训>（<主题缩写> <节>）`
- 推送：`git push origin main`（当前工作流直接在 main 上迭代）。
