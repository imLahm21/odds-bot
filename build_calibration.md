# build_calibration.py 使用说明

## 这脚本是什么

从本地 `竞彩.xlsx` 的下注战绩，重算一张「**自评置信度 → 真实胜率**」的校准表，
外加一张「玩法分类表现」表。输出是可直接粘进
[reference_staking_kelly.md](rules/方法论/reference_staking_kelly.md) 第二章的 Markdown。

它的作用：把你主观打的「置信度分」翻译成**历史上真实的胜率/ROI**，
从而定出「哪个置信度区间才值得下注」这种**慢变门槛**（如「≥70 才认真下、74+ 不加码」）。

> 只读 `竞彩.xlsx`，**不碰服务器、不碰赔率库、不改表格**。纯本地、几秒钟。

---

## 怎么跑

```bash
# 默认：统计用户选定的 5 个 sheet
python build_calibration.py

# 只取最近 3 个月（按 A 列日期滚动窗口）
python build_calibration.py --months 3

# 指定 sheet
python build_calibration.py --sheets 世界杯 20260701-20260731

# 指定表格路径 / 另存结果到文件
python build_calibration.py --xlsx 竞彩.xlsx --out calibration_out.md
```

默认统计的 5 个 sheet：
`20260409-20260502`、`20260501-20260531`、`20260520-20260630`、`20260701-20260731`、`世界杯`。

**参数：**

| 参数 | 作用 |
|------|------|
| `--sheets` | 指定统计哪些 sheet（空格分隔多个）；默认上面 5 个 |
| `--months N` | 只取最近 N 个月的行（按日期滚动窗口）；缺日期的行仍计入 |
| `--xlsx` | 战绩表路径，默认 `竞彩.xlsx` |
| `--out` | 把 Markdown 另存到文件（默认只打印到终端） |

## 多久跑一次（**不是每天**）

**低频重跑即可**：

- **每新增约 30 笔下注**，或**每周一次**，重跑一次。
- **改了 SOP 判读规则后**（你的「置信度尺子」变了），立即跑一次看校准有没有漂移。

**为什么不用每天跑**：单日通常只有几笔，对「70–74 桶胜率」这种统计量的影响是噪声——
今天加 3 笔让某桶从 67.4% 抖到 67.0%，不值得每天改文档、重启 bot。
统计校准要样本累积到肉眼可见的变化才有意义。

---

## 校准表 vs p_市场 —— 两层，不同频率

| | 更新频率 | 谁在用 | 定什么 |
|---|---|---|---|
| **校准表**（本脚本产出） | 每 1–2 周 / 每周 | 慢变门槛 | 「≥70 才下、74 封顶、让球降级」 |
| **p_市场**（LLM 精算时现场去抽水） | **每场实时** | 每一场 | 「这场每个玩法的 edge 和注额」 |

**每天「这场买什么、买多少」靠 p_市场，它由 bot 每次 `/analyze` 实时算、天然最新，不依赖本脚本。**
校准表只是给 p_市场 加一个「你在哪个置信度区间可信」的慢变权重系数，一两周更新一次足够。

> 换句话说：**不重跑脚本 ≠ 决策过时**。日常决策的新鲜度由 p_市场 保证，校准表只管慢变的能力圈门槛。

---

## 跑完做什么

1. 把终端输出的两张 Markdown 表，**替换** `reference_staking_kelly.md` 第二章里的旧表。
2. **重启 tg bot**（规则文件是进程内缓存 `_rules_cache`，重启才会重新加载）。
3. 完成。下次 `/analyze` 精算时，LLM 就用上了最新校准门槛。
